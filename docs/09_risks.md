# 主要风险和替代方案

在项目开始前就把可能的风险想清楚，遇到问题时不慌。

---

## 风险 1：Lua 学习曲线导致进度慢

### 风险描述
你不懂 Lua，也不懂游戏开发，前期可能因为语言特性和 LÖVE2D API 不熟悉导致进度慢。
特别是 Lua 的一些坑（闭包自引用、1-based 索引、falsy 值判断）可能在调试上浪费时间。

### 严重程度
**中** —— 你有多种语言经验，学习 Lua 语法本身很快。但"知道语法"和"写出地道的 Lua 代码"之间有差距。

### 应对方案
1. **V0 脚手架代码我帮你写好**：带详细注释，每个 Lua 特性都有类比说明
2. **代码规范文档**：`docs/05_code_style.md` 列出了所有常见坑和最佳实践
3. **遇到问题先查规范**：规范里有的直接按规范来，不用自己想
4. **迭代式开发**：每个迭代范围小，出问题影响面也小
5. **多写注释**：你写的代码也加注释，未来的你会感谢现在的你

### 替代方案
如果 Lua 实在太别扭，可以考虑：
- **用 moonscript**（编译到 Lua 的语言，语法类似 CoffeeScript）—— 不推荐，增加复杂度
- **换引擎**：Godot (GDScript)、Phaser (JS) —— 但就失去了 LÖVE2D 的轻量和代码控制感

**结论**：坚持用 Lua。你有编程基础，两周内就能上手，一个月就能熟练。

## 风险 2：架构设计过度，做了一堆用不上的东西

### 风险描述
因为想做"可复用的通用引擎"，可能在 core 层堆了很多功能，但实际做中国象棋根本用不上。
为了"未来扩展性"牺牲了当下的开发效率。

### 严重程度
**高** —— 这是独立开发者最容易犯的错：过度工程化（over-engineering）。

### 应对方案
1. **V0 只做最必要的 8 个核心模块**：不多做
2. **YAGNI 原则**：You Aren't Gonna Need It. 真的需要的时候再加
3. **每个版本的范围严格控制**：比如 V2 就专注于下棋逻辑，不加别的
4. **core 层功能由 game 层需求驱动**：game 层需要什么，core 层才加什么
5. **定期复盘**：每个迭代结束想想，哪些东西做了但没用

### 替代方案
如果发现 core 层太复杂了：
- **简化**：把一些模块合并，去掉不必要的抽象
- **回退**：V1 之后如果觉得分层没必要，可以把代码揉在一起
- **接受不完美**：第一个游戏就是踩坑的，架构不完美也没关系，下一个游戏再优化

## 风险 3：伪 3D / 纵深感效果成本太高

### 风险描述
你想要"2D 但有一点纵深感"的效果。如果用了太复杂的技术（比如 2.5D、透视投影、多层视差 + 光照），
可能开发成本远超预期，而且效果不一定好。

### 严重程度
**中高** —— 视觉效果是个无底洞，可以无限优化。

### 应对方案
1. **V0-V3 先不加纵深效果**：先做平面的，能玩最重要
2. **V4 再加简单的纵深感**：用以下低成本方式（按成本从低到高排序）：
   - **阴影**：棋子下面加个半透明的圆阴影——成本极低，效果明显
   - **棋子选中时上浮**：y 坐标往上挪一点 + 阴影变大——简单有效
   - **多层背景**：背景分 2-3 层，移动时速度不同（视差）——成本中等
   - **粒子效果**：选中/吃子时的粒子——用粒子系统，不用自己写
   - **颜色渐变**：棋盘从远到近颜色微微变深——暗示纵深感
3. **避免做这些**（成本高效果一般）：
   - ❌ 真 3D 透视投影
   - ❌ 复杂的光照系统
   - ❌ 大量精灵分层模拟 3D

### 替代方案
如果觉得纵深感没必要或者太麻烦：
- **纯平面风格**：也可以很好看，参考极简风格游戏
- **像素风格**：像素风天然不需要纵深感
- **手绘风格**：用美术质感弥补深度感

## 风险 4：中国象棋规则复杂，容易有 bug

### 风险描述
中国象棋的规则看起来简单，但细节很多：蹩马腿、塞象眼、将帅不能对面、炮的吃子方式、士象的活动范围……
如果规则有 bug，游戏体验会很差。

### 严重程度
**中** —— 规则是游戏的核心，有 bug 不行。但规则是纯逻辑，可以写测试。

### 应对方案
1. **V2 先做基本走法**：每个棋子的移动规则，不考虑将军/将死
2. **V3 再加将军/胜负判定**：分阶段实现
3. **写单元测试**：规则部分是纯逻辑，可以写测试用例覆盖
4. **对照标准规则**：找权威的象棋规则文档对照
5. **多下棋测试**：你自己多玩，有意识地测试各种边界情况

### 测试用例清单（V2 阶段列）
- 车：能不能越子、能不能斜走
- 马：蹩马腿的 8 种情况
- 象：塞象眼、能不能过河
- 士：能不能出九宫
- 将：能不能出九宫、将帅对面
- 炮：隔子吃、不隔子能不能走
- 兵：过河前后的走法

## 风险 5：AI 难度和性能平衡不好做

### 风险描述
AI 太简单了没意思，太复杂了性能有问题。而且中国象棋的 AI 算法（极小化极大 + alpha-beta 剪枝 + 估值函数）
需要一定的算法功底。

### 严重程度
**中** —— AI 是 V4 的内容，前期不用操心。但如果做得不好，单人模式体验会差。

### 应对方案
1. **分三个难度等级**：
   - **简单**：随机走合法步 + 优先吃子 —— 非常简单，让新手有成就感
   - **中等**：2-3 层搜索 + 简单估值 —— 大部分玩家的水平
   - **困难**：4-5 层搜索 + 完善估值 —— 有挑战
2. **V4 先做简单和中等**：困难版以后再加
3. **参考开源实现**：中国象棋 AI 有很多开源实现，可以借鉴思路
4. **用 LuaJIT**：LÖVE2D 自带 LuaJIT，性能比标准 Lua 好很多

### 替代方案
如果 AI 太难做：
- **只做双人模式**：也可以发布，很多玩家就是想和朋友一起玩
- **找开源 AI 集成**：比如用 FFI 调用 C 写的象棋引擎
- **后续再补**：先发布，AI 作为后续更新内容

## 风险 6：Steam 发布不顺利，卖不动

### 风险描述
花了几个月做出来的游戏，上架 Steam 后没人买。
中国象棋在 Steam 上是小众品类，而且有免费竞品。

### 严重程度
**中高** —— 但这是 V5 之后才需要担心的事。

### 应对方案
1. **降低预期**：第一个游戏不追求赚多少钱，主要是练手和验证流程
2. **定价策略**：低价（比如 6-18 元），走量
3. **差异化**：
   - 画面品质比免费象棋好
   - 有 AI 难度分级
   - 有棋谱/复盘功能
   - 支持创意工坊（玩家自制棋盘皮肤）
4. **营销**：
   - 发试玩版
   - 在相关社区宣传（象棋吧、Steam 社区等）
   - 找 UP 主/主播试玩
5. **V5 之后再考虑**：先把游戏做好，发布的事成到时候再说

### 替代方案
如果 Steam 不行：
- **itch.io**：独立游戏平台，上架简单，分成友好
- **免费+内购**：但做起来麻烦
- **纯个人项目**：不卖，就当练手和作品集

## 风险 7：项目烂尾

### 风险描述
独立开发最常见的风险：做着做着没动力了，或者被别的事情打断，项目就烂尾了。

### 严重程度
**高** —— 统计上 90% 的个人项目都会烂尾。

### 应对方案
1. **迭代式开发**：每个版本都有可玩的成果，持续有正反馈
2. **小步快跑**：每个迭代 1-2 周，不会太长
3. **先做核心玩法**：V2 就能下棋了，就算烂尾在 V2，也有个能玩的东西
4. **设定明确的里程碑**：每个版本有明确的验收标准
5. **保持节奏**：不用天天做，但要保持频率（比如每周至少 2 次）
6. **记录进度**：看到自己做了这么多东西，会有动力继续

### 烂尾也能接受的底线
就算只做到 V3，也有：
- 一个可复用的 2D 游戏引擎脚手架
- 一个能完整下一局棋的中国象棋游戏
- 学会了 Lua 和 LÖVE2D
- 积累了独立游戏开发的经验

这些都是收获，不算白做。

## 风险 8：远程仓库推送问题

### 风险描述
你提到"推送到远程仓库，名称为 trillion_games_2D_brich"，但没有给具体的仓库 URL 和平台。
可能是 GitHub、Gitee、GitLab，或者自建仓库。

### 严重程度
**低** —— 不影响开发，只是推送的时候会遇到。

### 应对方案
1. **V0 阶段先初始化本地 Git 仓库**
2. **打 tag 在本地**
3. **等你提供远程仓库 URL 后再推送**
4. **如果还没创建远程仓库**：建议用 GitHub 或 Gitee，创建好后给我 URL

### 需要你提供的信息
- 远程仓库的完整 URL（例如 `https://github.com/yourname/repo.git`）
- 认证方式（SSH key 还是用户名密码）
