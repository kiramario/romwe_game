# 文档规范

## 为什么要有文档

- 代码告诉你"怎么做"，文档告诉你"为什么这么做"
- 三个月后的你回来，需要文档快速回忆上下文
- 架构决策、版本规划、发布流程这些东西不适合放在代码注释里

## 文档分类

### 1. 设计文档（docs/ 下的 md 文件）
- 项目愿景、架构、路线图、规范
- 这类文档是"宏观文档"，在项目初期写好
- 重大架构变更时更新

### 2. 代码注释（代码内）
- 函数注释、行内注释
- 这类文档是"微观文档"，写代码的时候写
- 改代码时同步更新

### 3. API 文档（docs/07_modules.md）
- 核心模块的公开 API 说明
- 类似 JavaDoc 或 JSDoc
- 模块稳定后写，模块变更时更新

### 4. 操作手册（tools/ 目录的 README）
- 构建脚本、发布脚本的使用说明
- 运维类文档

## 设计文档规范

### 文件名
- 数字前缀 + 下划线 + 简短名称
- 例如：`00_vision.md`, `01_architecture.md`
- 数字前缀确保文件按顺序排列

### 结构
每个设计文档尽量包含：
1. **标题**：一句话说明文档讲什么
2. **正文**：分章节，用 ## 二级标题
3. **版本信息**：文档末尾标注版本和更新日期

### Markdown 规范
- 标题层级：# 一级（文档标题），## 二级（章节），### 三级（小节）
- 列表用 - 号，不用 * 号
- 代码块标注语言：` ```lua `
- 表格对齐：冒号在左边左对齐，两边冒号居中

## 代码注释规范

### 什么该写注释
- ✅ 为什么这么做（设计决策）
- ✅ 复杂算法的思路
- ✅ 容易踩坑的地方（比如 Lua 的闭包自引用问题）
- ✅ 公开 API 的参数和返回值
- ❌ 显而易见的事情（`x = x + 1 -- x 加 1`）
- ❌ 翻译代码为中文（用代码本身表达意图）

### 文件头注释
每个 Lua 文件开头加注释：
```lua
-- 模块名: xxx
-- 功能: 简要描述这个模块做什么
-- 依赖: 依赖哪些其他模块
```

### 函数注释
公开函数必须有注释：
```lua
-- 获取指定位置的棋子
-- @param x (number) 棋盘 x 坐标 (1-9)
-- @param y (number) 棋盘 y 坐标 (1-10)
-- @return (table|nil) 棋子对象，没有则返回 nil
function Board:get_piece(x, y)
end
```

### 行内注释
复杂逻辑旁边加注释，解释"为什么"而不是"做什么"：
```lua
-- 用 table.concat 而不是 .. 拼接，避免循环中产生大量临时字符串
local result = table.concat(lines, "\n")
```

## 文档更新原则

### 什么时候更新文档
1. **架构变更时**：加了新模块、改了依赖关系
2. **版本发布时**：更新路线图、版本日志
3. **踩坑后**：把坑记下来，避免再踩
4. **需求变化时**：愿景、范围变了要更新

### 不要追求文档完美
- 文档够用就行，不用写论文
- 宁愿有一个 80 分但及时的文档，也不要一个 100 分但永远写不完的文档
- 代码是真相的最终来源，文档是辅助理解的地图
