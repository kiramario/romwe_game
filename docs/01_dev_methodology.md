# 开发方式选型：迭代式开发（Iterative Development）

## 推荐方式：迭代式开发

在"瀑布、螺旋、迭代、增量、原型、RAD、JAD、敏捷、XP、Scrum、精益、DevOps"中，
**最适合独立开发者 + LÖVE2D + 中国象棋项目**的开发方式是：**迭代式开发**。

## 为什么选迭代式，不选其他

### ✅ 迭代式开发 (Iterative Development) —— 推荐

**核心理念**：从一个简单的初始版本开始，通过多次迭代逐步丰富功能。
每次迭代都产出一个可运行、可测试的完整版本。

**为什么适合这个项目**：
1. **你是独立开发者**：不需要复杂的流程框架（Scrum 站会、XP 结对编程都不适用）。
2. **你在学新东西**：Lua + LÖVE2D 都是新领域。迭代式允许你边做边学，早期犯的错可以在后续迭代中修正。
3. **架构需要验证**：core/game 分层、可复用性这些架构假设需要通过实际开发来验证。
4. **每个版本都有成就感**：每个版本打 tag，随时可切换测试，符合你的工作习惯。
5. **风险可控**：每个迭代范围小，出问题容易回退。

### ❌ 为什么不选其他

| 方式 | 不选原因 |
|------|----------|
| **瀑布** | 需求到交付一次性走完，中间不能改。你在学新技术，不可能一开始就把所有设计想对。 |
| **螺旋** | 强调风险分析和原型循环，太重了。独立开发者不需要这么重的流程。 |
| **增量** | 先做完整架构再一块块加功能。但你不确定架构对不对，增量式容易在错误架构上越建越多。 |
| **原型** | 原型只是验证，不强调演进为最终产品。我们的 V0 就要是可演进的正式脚手架。 |
| **RAD / JAD** | 强调快速工具和用户紧密协作。你既是开发者也是用户，不需要专门的 JAD 流程。 |
| **敏捷** | 太泛了，是一套价值观不是具体方法。迭代式是敏捷价值观下的具体做法。 |
| **XP** | 强调结对编程、测试驱动、持续集成。单人项目用不上结对编程，TDD 对游戏开发也不友好。 |
| **Scrum** | 需要产品负责人、Scrum Master、团队、Sprint 规划会。你一个人就是整个团队，用 Scrum 是自寻烦恼。 |
| **精益** | 强调 MVP 和 validated learning。理念可以借鉴，但不是一套开发流程。 |
| **DevOps** | 是开发+运维的文化，不是开发方法论。我们的发布流水线会用到 DevOps 工具，但这不是开发方式。 |

## 迭代式开发在本项目的具体实践

### 每个迭代的固定节奏

```
规划 → 编码 → 测试 → 打tag → 发布 → 复盘 → 下一迭代
```

### 每个迭代的原则

1. **时间盒优先于范围**：每个迭代有大致时间范围（比如 1-2 周），宁可砍功能也要按时产出可玩版本。
2. **每个版本必须可运行**：不能有"半成品"版本。打 tag 的版本必须能启动、能看到明确成果。
3. **自上而下设计，自下而上实现**：架构先想清楚（core/game 分层），但实现从小处开始（V0 只跑通最基础链路）。
4. **每个迭代复盘**：完成后花 30 分钟想想：这一版哪些做得好，哪些要调整，下一版要不要改方向。

### 与版本规划的对应

- V0 ~ V5 是 6 个迭代
- 每个迭代有明确的目标、范围、验收标准
- 每个迭代结束打 tag，可随时切换回测

## 借鉴其他方式的精华

虽然主选迭代式，但我们从其他方式中借鉴好的实践：

- **从精益借鉴**：每个版本都是一个 MVP 假设，通过"玩"来验证
- **从 DevOps 借鉴**：自动化构建、打包脚本、一键发布
- **从原型法借鉴**：不确定的交互先用快速原型验证，再正式实现
- **从增量式借鉴**：架构设计有前瞻性，不是完全走一步看一步
